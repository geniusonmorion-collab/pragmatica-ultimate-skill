# Контракт доски — файлы `research/` и готовые шаблоны

Доска — единственный источник правды проекта. Она живёт **в рабочем проекте клиента**: папка `research/` рядом с файлами проекта — НЕ в репозитории методологии и не в домашней папке скилла.

Правила создания:

- Создавай `research/` при первом запуске скилла в проекте (стартовые инварианты). Если папка уже есть — читай и пополняй, никогда не пересоздавай и не перезаписывай без диффа.
- Пять файлов создаются сразу (пустой по шаблону — тоже файл); `probes/` и `visual-corpus/` — по мере надобности.
- Всё в файлах — на языке дизайнера: служебных слов только девять, из глоссария (см. `language.md`). Эти файлы читают люди, не только ты.

| Файл | Что хранит | Когда создаётся |
|---|---|---|
| `research/goal-card.md` | карточка «зачем этот проект» | первый запуск, до любых артефактов |
| `research/ledger.yaml` | реестр ставок (≤25 живых) | первый запуск |
| `research/board.md` | человекочитаемая доска: глоссарий, карта бизнеса, находки ходов, нестыковки, морозильник + полка зацепок, бюджет | первый запуск |
| `research/probe-map.yaml` | план проверок: носитель + исполнитель + порог | первый запуск |
| `research/events.md` | журнал звоночков (append-only) | первый запуск |
| `research/probes/<id-ставки>--<носитель>/` | пробы-носители: `index.html` + `probe.yaml` | по мере надобности |
| `research/visual-corpus/` | корпус экранов + `corpus-index.yaml` | по мере надобности |

Правила письма — общие для всех файлов:

- Ставка = глагол + конкретика, ≤25 слов, без оценочных прилагательных.
- Цифра всегда с источником и датой; без ссылки — автоматически «предполагаю».
- Автор на каждой записи: пишут все, «пиши, не спрашивай»; бюрократия только на убийстве ставок.
- Сказанное в чате, личке или на созвоне для проекта не существует, пока не стало записью в этих файлах.
- Каждый артефакт проекта несёт ставку-родителя: экран, проба, скрин корпуса без привязки к ставке — брак (проверяет `linter.md`).

## `research/goal-card.md` — карточка «зачем»

Заполняется до любых артефактов. Одна главная цель; двойная цель без приоритета — карточка невалидна. Шаблон — скопируй целиком:

```markdown
# Карточка «зачем этот проект» — <имя проекта>
Дата: · Автор: · Статус: черновик | подписана лидом

## Лестница «зачем» (3–5 ступеней вверх от брифа)
1. Бриф просит: …
2. Зачем это клиенту: …
3. Зачем это ЕМУ: …
4. …
(остановись, когда следующая ступень уже не про этот проект)

## Одна главная цель
> …
(ровно одна; «и конверсия, и бренд» без приоритета — карточка невалидна)

## Развилка: полируем текущее или меняем подход
Выбор: полируем | меняем · Почему: …

## Отрезанные альтернативы
- … (что решили НЕ делать, выбрав эту цель, — и почему)
- …

## Где живёт проект — флажок на карте бизнеса
Узел: …
(для нового продукта флажок стоит на «кто платит и за что»: проверка этого узла —
мандат проекта, а не повод для паники)

## Флаги контекста
- [ ] живой продукт
- [ ] платящая база
- [ ] B2B
- [ ] регулируемая отрасль / железо / госсистемы → сразу часовой скан ограничений

## Быстрые прогоны по брифу
- Задача за каждой названной «проблемой»: «проблема …» → на самом деле человек хочет …
- Чему в брифе можно верить (с меткой): …
- Кого бриф отрезал: …
- Наивные ставки-мишени: … (→ сразу кандидаты в ledger.yaml)
```

## `research/ledger.yaml` — реестр ставок

Живых (`status: alive`) — не больше 25. Написал 26-ю — сначала убей, замри в морозильник или объедини. Ставка без `dies_if` или без `test` — не ставка, а мнение: живёт как `candidate`, в `alive` не переводится.

Схема — скопируй шапку и веди список:

```yaml
# research/ledger.yaml — реестр ставок проекта
# Правила: ≤25 живых; text ≤25 слов, глагол + конкретика; «знаю» — только при url+date;
# несущая ставка (load_bearing) не строится на «тёплой выборке» или «реверсе без рук»
# и убивается только фактом того же веса — после Разбора провала, не мнением.
bets:
  - id: st-001            # st-NNN, сквозная нумерация, id не переиспользуются
    text: ""              # ≤25 слов, глагол + конкретика, без оценочных прилагательных
    for_whom: ""          # сегмент + задача пользователя; «все» — не сегмент
    author: ""            # кто записал
    load_bearing: false   # несущая: на ней стоит цель, скоуп или «кто платит»
    confidence: предполагаю   # знаю | предполагаю | не знаю
    sample: ""            # пусто | тёплая выборка | реверс без рук — честная пометка происхождения
    evidence:             # лестница силы факта: кто-то сказал → цифры → опрос →
      - url: ""           #   разговоры с плательщиками → попробовали → заплатили
        date: 2026-01-01
        note: ""          # что именно источник говорит + ступень лестницы
    dies_if: ""           # «неверна, если…» — наблюдаемое условие, не «если мы ошиблись»
    test:
      method: ""          # самый дешёвый тест
      threshold: ""       # порог «риск проявился / нет»; назначается ДО теста
      owner: ""
    feeds_decisions: []   # какие решения на ней стоят; пусто = ставка никому не нужна
    status: candidate     # candidate | alive | killed | validated
    expires: 2026-01-01   # срок годности; истёк → переоткрыть или в реестр рисков, молчать нельзя
```

Два заполненных примера (обезличены):

```yaml
bets:
  - id: st-001
    text: "Лиды гибнут на шаге заявки на сайте, а не после передачи менеджеру"
    for_whom: "малый бизнес, заполняющий заявку сам, без закупщика"
    author: "ведущий дизайнер"
    load_bearing: true
    confidence: предполагаю
    sample: "тёплая выборка"
    evidence:
      - url: ""
        date: 2026-06-30
        note: "слова директора по продажам на интейке — ступень «кто-то сказал»"
    dies_if: "по CRM ≥60% потерь происходят уже после передачи менеджеру"
    test:
      method: "час в CRM клиента: доля потерь до и после передачи, помесячно, по сегментам"
      threshold: "≥60% потерь до передачи — жива; иначе умерла"
      owner: "аналитик"
    feeds_decisions: ["структура главной страницы", "приоритет фичлиста: форма заявки"]
    status: alive
    expires: 2026-08-01

  - id: st-014
    text: "При повторном заказе люди меняют одно поле, а не собирают заказ заново"
    for_whom: "постоянные покупатели: ≥2 заказа за квартал"
    author: "дизайнер"
    load_bearing: false
    confidence: не знаю
    sample: ""
    evidence: []
    dies_if: "в аналитике ≥50% повторных заказов правят 3+ поля"
    test:
      method: "час в аналитике: сколько полей меняют при повторном заказе"
      threshold: "≤1 поле у ≥70% — жива (флоу А, один экран); иначе флоу Б, мастер"
      owner: "дизайнер"
    feeds_decisions: ["выбор между флоу А и Б", "фичлист онбординга"]
    status: alive
    expires: 2026-07-20
```

## `research/board.md` — человекочитаемая доска

Читается сверху вниз за 5 минут. Глоссарий девяти слов стоит в шапке и не убирается. Шаблон — скопируй целиком:

```markdown
# Доска проекта — <имя проекта>
Владелец: … · Обновлено: …

## Самое рискованное допущение сейчас
> st-…: … (вопрос каждого пульса: оно всё ещё то же?)

## Глоссарий — девять слов (один экран, не убирать из шапки)
| Слово | Значение в одну строку |
|---|---|
| Ставка | утверждение, за которое отвечаем делом: делаем / не делаем / рискуем |
| Знаю / предполагаю / не знаю | метка на каждом факте; «знаю» — только со ссылкой и датой |
| Aha-момент | момент, когда продукт впервые оправдал — или превысил — ожидания человека |
| Сегмент | люди с одинаковыми задачами и критериями «получилось» — не возраст и не должность |
| Барьер / страх | реально нельзя (чинит клиент) / кажется, что нельзя (чиним дизайном и текстом) |
| MVP-проба | минимальная вещь, чтобы риск проявился, — не чтобы продать |
| Юнит-экономика | сходится ли у клиента экономика одного заказа |
| Зацепка | наблюдение-странность до всякой идеи: факт дословно + «почему этого не должно быть» + догма под ударом |
| Козырь | правда, в которую конкуренты не верят или которую им дорого признать, + наша дешёвая проверка, + фора |

## Карта бизнеса клиента
(узел «кто платит и за что» — отдельной строкой на каждый сегмент; сегменты не усредняются)
| Узел | Что знаем — одной строкой, с источником | Метка | Наш проект живёт здесь? |
|---|---|---|---|
| Рынок: откуда приходят деньги | … | знаю/предполагаю/не знаю | |
| Кто платит и за какие задачи — сегмент «…» | … | … | 🚩 |
| Кто платит и за какие задачи — сегмент «…» | … | … | |
| Чем мы лучше | … | … | |
| Сходится ли экономика (юнит-экономика) | … | … | |
| Покупают и возвращаются ли | … | … | |

## Находки ходов (бенчи, аудиты, карты, пути, словарь и тон)
(сюда ложатся выходы ходов из `moves.md`; каждая находка несёт ставку-родителя (st-…)
и источник с датой — находка без адресата не записывается)

## Нестыковки (≤12 карточек, каждая ≤5 строк)
### Н-01 · тип: барьер | страх | конфликт приоритетов · статус: открыта | тест назначен | закрыта
- Источник А: «цитата дословно» (ссылка, дата)
- Источник Б: «цитата дословно» (ссылка, дата)
- Что разведёт: тест + порог (st-…) — или «барьер, чинит клиент» → эскалация

## Морозильник идей («не сейчас», не «мертво»)
| Идея | Чья задача (словами пользователя) | Почему не сейчас | Что должно стать правдой — звоночек-переоткрыватель | Автор |
|---|---|---|---|---|

## Полка зацепок (наблюдения-странности; сырьё Контура B)
| Наблюдение — цитата/цифра дословно (URL + дата) | Почему странно: «если верить X, этого не должно быть» | Догма под ударом | Сила факта | Окно до (для сдвигов) |
|---|---|---|---|---|

## Бюджет — валюта: часы дизайнера
Курс: 1 час дизайнера ≈ 3 часа агента ≈ один средний прогон.
Дедлайн ограничивает календарь, бюджет — часы. Каждое меню печатает остаток отсюда.
| Этап | Часы дизайнера: выделено / остаток | Часы агента: выделено / остаток |
|---|---|---|
```

## `research/probe-map.yaml` — план проверок

Тест без носителя и исполнителя назначен «на бумаге» и назначенным не считается. Порог назначается ДО генерации носителя и после не двигается. Несколько рисков на одну ставку склеиваются в один тест; одна проба может бить по нескольким ставкам. Строки инструментации из «Дерева владения метриками» («артефакт → событие», ход №31 в `moves.md`) живут здесь же — с `carrier: событие`.

```yaml
# research/probe-map.yaml — план проверок
probes:
  - probe: ""             # что проверяем, одной строкой
    carrier: протик       # карточки | скетч | протик | фейк-дверь | безбилетник | проба-B | событие
                          #   (шесть носителей — таблица хода №50 в moves.md; «событие» — инструментация)
    bets: [st-000]        # ставки-родители; проба без ставки = брак, не заводится
    threshold: ""         # порог «риск проявился / нет»: действие + число
    owner: ""             # исполнитель: агент (носитель) / дизайнер (сессии) / чужой тест
    status: planned       # planned | ready | running | passed | failed | closed
    date: 2026-01-01      # дата теста
    recruit_calendar: ""  # для юзертестов обязательно: даты сессий, рекрут по квал-вопросам
    mandate: null         # фейк-дверь и всё от имени бренда: «да» клиента + дата, ДО генерации
    alt_explanations: ""  # что ещё объясняет провал; заполняется на Разборе провала,
                          # вилка «плохой макет» проверяется первой
    verdict: ""           # после теста: «риск проявился / нет»; при провале — ссылка на Разбор
```

Пример:

```yaml
probes:
  - probe: "Человек доходит до первого готового отчёта без подсказки"
    carrier: протик
    bets: [st-007, st-012]
    threshold: "≥3 из 5 доходят до отчёта за ≤6 шагов сами"
    owner: "агент (носитель) + дизайнер (сессии и разбор)"
    status: ready
    date: 2026-07-15
    recruit_calendar: "5 слотов 15–16.07, рекрут по квал-вопросам st-007"
    mandate: null
    alt_explanations: ""
    verdict: ""
```

## `research/events.md` — журнал звоночков

Append-only: строки не редактируются и не удаляются, меняется только статус. Необработанный звоночек блокирует любое новое меню (инвариант 5). Провал теста никогда не убивает несущую ставку автоматически — сначала Разбор провала.

```markdown
# Журнал звоночков
Строка = дата · событие (наблюдаемое, дословно — не «этап начался») · какие ставки
переоткрывает · автор · статус. Статусы: **новый** → **разобран** (ссылка на меню или запись).
Автор не классифицирует и не ждёт реакции: записал — работаешь дальше; разбирает владелец доски.

| Дата | Событие | Переоткрывает | Автор | Статус |
|---|---|---|---|---|
| 2026-07-03 | Тест hero провалил порог: главное действие назвали 2 из 10 (порог ≥7) | st-004, st-011 | дизайнер | новый |
| 2026-07-02 | Клиент принёс вводную: «добавьте тариф для крупных» | st-001, карта аудиторий | лид | разобран → меню 02.07 |
```

## `research/probes/` и `research/visual-corpus/` — по мере надобности

**Правило общее: каждый артефакт несёт ставку-родителя.** Файл без привязки к ставке — брак (см. `linter.md`).

### `probes/<id-ставки>--<носитель>/`

Создаётся генератором проб (см. `moves.md`) на каждую пробу: `probes/st-007--протик/`. Внутри:

- `index.html` — один самодостаточный файл: стили и переходы внутри, внешних зависимостей ноль, серый скелет; счётчик шагов до Aha-момента и лог кликов встроены.
- `probe.yaml` — карточка пробы:

```yaml
# probes/st-007--протик/probe.yaml
bet: st-007
carrier: протик
threshold: "≥3 из 5 доходят до отчёта за ≤6 шагов сами"
test_date: 2026-07-15
mandate: нет            # есть (дата + кто дал) / нет; для фейк-двери «нет» = брак
accent: нет             # один цветовой акцент разрешён только если ставка про заметность — тогда «да: …»
verdict: ""             # пусто до теста; после — «риск проявился / нет» + дата
click_log: ""           # выгрузка встроенного лога после сессий — сырьё Разбора провала
```

Протик — расходник теста: после вердикта файл остаётся в `probes/` следом, в дизайн не переезжает, не эволюционирует и не полируется. Выжившее дизайнер переносит руками в настоящие макеты — этот перенос и есть граница «проба / дизайн».

### `visual-corpus/<категория>/<приложение>--<экран>--<дата>.png` + `corpus-index.yaml`

Корпус экранов (Mobbin и скриншоты бенча). Лимит ≤7 находок на ход; скрин без строки индекса — факт без источника, бракуется. Корпус — рабочий материал проекта: не публикуется и не передаётся клиенту как библиотека.

```yaml
# research/visual-corpus/corpus-index.yaml
screens:
  - file: "банки/appname--онбординг--2026-07-01.png"
    app: ""
    url: ""
    version_date: 2026-05-14   # скрин = «наблюдение интерфейса с датой» — знаю, КАК выглядит;
                               # работает ли паттерн — не знаю: на эффект нужна ставка с тестом
    pattern: ""                # что за паттерн/код категории
    linked_to: st-000          # ставка / код категории / зацепка, на которой скрин стоит
```
